Getting Started

  1. Install the package: pip install -e ".[dev]"
  2. Copy example configs: copy config\models.example.json config\models.json and copy config\agents.example.json config\agents.json
  3. Run the onboard wizard: kinetic-cli onboard or edit config/models.json manually
  4. Set your API keys in .env file (see .env.example)
  5. Start the system: kinetic
  6. Open http://localhost:18789 in your browser

Chat

The Chat tab lets you send messages to your configured agent and see responses in real-time.

  • Session selector — Switch between different conversation contexts. Each session has its own message history and memory.
  • Send messages — Type in the input box and press Enter (or click Send). Use Shift+Enter for multiline messages.
  • Agent response — The agent uses its configured model and tools to respond. Tool execution (web search, file ops, etc.) happens automatically.
  • Tips: Ask the agent to search the web, save files, schedule reminders, or query its knowledge base.

Models Configuration

The Models tab lets you edit config/models.json directly. This file defines your LLM providers and stage routing.

File Structure

{
  "mode": "single|multi",
  "embedding": {
    "provider": "provider-name",
    "model": "text-embedding-3-small"
  },
  "defaults": {
    "classify":  { "provider": "...", "model": "..." },
    "think":     { "provider": "...", "model": "...", "fallbacks": [...] },
    "tool_call": { "provider": "...", "model": "..." },
    "answer":    { "provider": "...", "model": "..." }
  },
  "providers": {
    "my-provider": {
      "baseUrl": "https://api.example.com/v1",
      "apiKeyEnv": "MY_API_KEY"
    }
  }
}

Mode

  • single — One model handles everything (think stage only). Simpler, faster.
  • multi — Classify → Think → Tool_call → Answer. Each stage can use a different provider/model for cost optimization.

Adding a New Provider

  1. Add an entry under "providers" with a name of your choice (e.g., "openai", "groq", "ollama")
  2. baseUrl — The OpenAI-compatible API endpoint (must end in /v1)
  3. apiKeyEnv — The environment variable name (not the key itself) that holds your API key
  4. Set the corresponding env var in your .env file
  5. Reference the provider name in the "defaults" stages above

Stage Routing

Each stage serves a different purpose:

StagePurposeRecommended Model Type
classifyIntent classification (chitchat vs task)Small/fast (Qwen 1B, Gemma 1B)
thinkMain reasoningPowerful (Llama 70B, Claude, GPT-4o)
tool_callStructured tool outputTool-capable (Llama 3, GPT, DeepSeek)
answerResponse formattingSame as think, or cheaper

Each stage entry can have fallbacks — an array of { provider, model } to try if the primary fails.

Setting Up Embedding (RAG / Knowledge Base)

To use knowledge base features (index documents, semantic search, RAG), you need an embedding provider:

  1. Choose an embedding provider — This is separate from your chat model. Common options:
    • Ollama (local, free): nomic-embed-text or all-minilm
    • OpenAI: text-embedding-3-small or text-embedding-3-large
    • Any provider that offers /v1/embeddings endpoint
  2. Add the provider to the "providers" section if not already present: 
    "ollama": {
  "baseUrl": "http://127.0.0.1:11434/v1",
  "apiKeyEnv": "OLLAMA_API_KEY"
}
  3. Add the "embedding" section at the top level of models.json: 
    "embedding": {
  "provider": "ollama",
  "model": "nomic-embed-text"
}
  4. Set the env var in .env — for local providers like Ollama, use a dummy value: 
    OLLAMA_API_KEY=ollama
  5. Restart kinetic — You should see [RAG] Embedding ready in the logs
  6. If using Ollama, make sure the embedding model is pulled: 
    ollama pull nomic-embed-text
  7. Verify — Go to the Knowledge tab and inject a document. Then ask the agent a question about it.

Agents

The Agents tab lets you edit config/agents.json. This file registers agents in the system.

File Structure

{
  "settings": {
    "defaults": { "type": "library", "can_delegate": false }
  },
  "registry": [
    {
      "id": "main",
      "name": "Main Agent",
      "soulPath": "./main/SOUL.md",
      "provider": "openrouter",
      "model": "gpt-4o-mini",
      "can_delegate": true
    }
  ]
}

Fields

FieldDescription
idUnique identifier for the agent
nameDisplay name
soulPathPath to the SOUL.md file (relative to config/) that defines the agent's system prompt / personality
providerWhich provider to use (must match a key in models.json)
modelModel to use (if omitted, uses the think stage default)
can_delegateWhether this agent can spawn sub-agents for specialized tasks

SOUL.md — Agent Personality

Each agent has a config/<agent-id>/SOUL.md file that serves as its system prompt. This is where you define:

  • Personality — Who the agent is and how it should behave
  • Communication style — Concise, formal, friendly, etc.
  • Core directives — Rules the agent must follow (e.g., "Never state 'As an AI language model'")
  • Boundaries — What the agent can and cannot do

SOUL.md changes take effect immediately on restart — the system detects stale prompts and refreshes them automatically.

Tips

  • You can have multiple agents with different personalities (e.g., main + code_expert + researcher)
  • Agents with can_delegate: true can spawn ephemeral specialist agents at runtime
  • Save via the dashboard then restart kinetic for changes to take effect

Knowledge Base

The Knowledge tab lets you manage the RAG (Retrieval-Augmented Generation) system. Requires an embedding provider to be configured in models.json (see Models docs above).

Injecting Documents

You can add content to the knowledge base three ways:

  • Text — Paste or type content directly
  • URL — Fetch a webpage and index its content
  • File Path — Index a file from the sandbox (relative to project root)

The system automatically chunks the content, generates embeddings, and stores them in agents_workspace/<agentId>/knowledge/store.json.

Querying

Ask the agent questions about your documents. The agent will automatically search the knowledge base when relevant. You can also use the /knowledge commands in Telegram.

Storage

All knowledge is stored as a JSON file on disk — no external database needed. Each document is chunked, embedded with cosine similarity + keyword hybrid search for retrieval.

Sessions

Sessions let you maintain multiple independent conversation contexts. Each session has its own:

  • Message history (JSONL file at agents_workspace/<agentId>/history.jsonl)
  • User profile (extracted every 3 messages)
  • Memory and context

Managing Sessions

  • View sessions — See all existing sessions with the active one highlighted
  • Switch sessions — Click "Switch" to change context
  • Create sessions — Type a name and click "Create" for a fresh conversation
  • History cap — Each session is capped at 500 messages. Oldest messages are trimmed first (system prompt always kept).

You can also switch sessions from the Chat tab's session dropdown.

Pipelines

Pipelines let you chain multiple agent calls in sequence, passing outputs between steps. Useful for multi-stage processing workflows.

Structure

{
  "name": "My Pipeline",
  "description": "What this pipeline does",
  "steps": [
    {
      "agentId": "main",
      "prompt": "Summarize this text: {{input_text}}",
      "outputVar": "summary"
    },
    {
      "agentId": "main",
      "prompt": "Translate to French: {{summary}}",
      "outputVar": "translation"
    }
  ]
}

How It Works

  1. Each step sends its prompt to the specified agent
  2. {{variable}} placeholders are replaced with outputs from previous steps
  3. The outputVar stores that step's result for use in subsequent steps
  4. Pipeline outputs are returned as a JSON object with all variable names

Execution

From the Pipelines tab, click "Run" on any pipeline. You can also execute pipelines via Telegram (/pipeline <name>) or CLI (kinetic-cli pipelines execute <id>).
